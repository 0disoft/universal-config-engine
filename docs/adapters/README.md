## Product Boundary

Universal Config Engine does not parse every config dialect directly. Parser
adapters feed normalized config values into the pipeline. Validator adapters inspect
the normalized config object and report normalized validation issues.

These examples are contract fixtures, not built-in support claims. YAML, TOML, INI,
and JSON5 remain outside core unless a future ADR adds first-party package
ownership, parser dependencies, fixtures, and release gates.

## Loader Adapter Contract

A loader adapter returns the same shape as `ConfigLoaderResult`:

- `value`: normalized config value accepted by core;
- `locations`: optional value-to-source locations;
- `issues`: optional bounded source, parse, mapping, validation, redaction, or
  resource-limit issues.

The loader descriptor must keep source identity stable:

- `id`: stable adapter source id;
- `kind`: `adapter` for third-party parser examples;
- `priority`: explicit merge priority;
- `displayName`: human-readable source name;
- `redaction`: optional secret metadata.

Adapter failures must become `ConfigIssue` objects. Throwing from a loader is still
normalized by core as `source-load`, but parser adapters should prefer explicit
`parse` issues when the parser can identify a dialect failure.

Core never copies a thrown loader exception message into diagnostics. Parser
adapters must also avoid copying provider exception text or malformed raw input
into issue messages or details.

## Validator Adapter Contract

A validator adapter receives normalized config:

- `config`: the resolved normalized config object;
- `provenance`: resolution provenance generated by core.

It returns:

- `ok`: validator status;
- `value`: optional adapter-local typed output;
- `issues`: structured `ValidatorIssue` values containing `code`, `severity`, and
  an optional normalized config `path`.

`value` is not written back to the pipeline output. Later validators still receive
the resolved config produced by core. ADR 0009 owns this boundary.

Validator exceptions are normalized without copying exception text into diagnostic
output. Core also ignores adapter-provided message, details, category, and source id
fields. It reconstructs the public `ConfigIssue` with the registered validator id
and a stable generic message. ADR 0011 owns this trust boundary.

JSON Pointer adapters must preserve numeric-looking object keys. The first-party Ajv
adapter converts a segment to a numeric index only when traversal of the validated
instance shows that its parent is an array; ambiguous helper calls retain strings.

## Fixture Rules

Fixtures under `docs/adapters/fixtures/` are checked by
`pnpm run check:adapter-fixtures`.

- `*.loaded-source.json` models a parser adapter output after normalization.
- `*.validator-result.json` models a validator adapter result.
- Fixture paths use normalized config paths, not raw parser token paths.
- Fixture issues must be bounded and must not include raw secret values.
- Loader fixture source ids must match the descriptor they document.
- Validator result fixtures contain only structured validator issue fields.

## Current Fixtures

- `yaml-loader.loaded-source.json`: location-aware parser adapter success.
- `json5-loader.loaded-source.json`: parser adapter success with JSON5-style input
  normalized to ordinary config values.
- `toml-loader-parse-failure.loaded-source.json`: parser adapter parse failure.
- `validator-path.validator-result.json`: validator issue paths against normalized
  config.

## Current Examples

`docs/adapters/examples.md` shows parser-loader and validator-path adapter patterns
for YAML, TOML, JSON5, and JSON Pointer-style validator errors. These examples use
adapter-supplied parser and validator functions; they do not add parser packages to
core or claim built-in dialect support.
